### Bloque 5: Agrupaciones Multitabla (INNER, LEFT, RIGHT + GROUP BY)

**21. Calcula el importe total gastado por cada cliente. Muestra el nombre completo y la suma de los subtotales de sus compras (usa `INNER JOIN`).**

SQL

```sql
SELECT c.nombre_completo, SUM(lp.subtotal) AS total_gastado
FROM clientes c
INNER JOIN pedidos p ON c.id = p.cliente_id
INNER JOIN lineas_pedido lp ON p.id = lp.pedido_id
GROUP BY c.nombre_completo;
```

**Explicación:** Unimos las tres tablas para llegar desde el cliente hasta el subtotal de cada línea. Como queremos un total "por cada cliente", ponemos su nombre en el `SELECT` y en el `GROUP BY`, y sumamos con `SUM()`.

📝 **Chuleta:** Al usar `INNER JOIN`, los clientes que se registraron pero no tienen pedidos (como Pedro Acosta) desaparecerán automáticamente de este reporte.

**22. Cuenta cuántos pedidos ha realizado cada cliente. Muestra el nombre y la cantidad.**

SQL

```sql
SELECT c.nombre_completo, COUNT(p.id) AS cantidad_pedidos
FROM clientes c
INNER JOIN pedidos p ON c.id = p.cliente_id
GROUP BY c.nombre_completo;
```

**Explicación:** Agrupamos por cliente y usamos `COUNT(p.id)` para contar cuántos identificadores de pedido únicos tiene asociados.

📝 **Chuleta:** Podrías usar `COUNT(*)` aquí porque al ser un `INNER JOIN` no hay filas nulas arrastradas, pero acostúmbrate a contar la clave primaria de la tabla "hija" (`p.id`) para evitar sustos.

**23. Modifica la consulta anterior para que aparezcan todos los clientes, incluso los que tienen 0 pedidos (usa `LEFT JOIN` y `COUNT(pedidos.id)`).**

SQL

```sql
SELECT c.nombre_completo, COUNT(p.id) AS cantidad_pedidos
FROM clientes c
LEFT JOIN pedidos p ON c.id = p.cliente_id
GROUP BY c.nombre_completo;
```

**Explicación:** El `LEFT JOIN` fuerza a que salgan todos los clientes. Si un cliente no tiene pedidos, `p.id` será nulo. `COUNT(p.id)` ignora los nulos, por lo que devolverá un `0` perfecto para esos clientes.

📝 **Chuleta:** ¡Cuidado! Si aquí usaras `COUNT(*)`, la base de datos contaría la "fila entera" que ha devuelto el `LEFT JOIN`, y los clientes sin pedidos saldrían con un `1` (falso) en lugar de un `0`.

**24. Calcula cuántas unidades en total se han vendido de cada producto. Muestra el nombre del producto y la suma de cantidades.**

SQL

```sql
SELECT p.nombre, SUM(lp.cantidad) AS total_unidades
FROM productos p
INNER JOIN lineas_pedido lp ON p.id = lp.producto_id
GROUP BY p.nombre;
```

**Explicación:** Unimos el catálogo con las ventas. El `GROUP BY` agrupa las ventas por producto, y el `SUM()` acumula el valor numérico de la columna `cantidad`.

📝 **Chuleta:** `SUM()` suma valores dentro de una columna. `COUNT()` cuenta cuántas filas existen.

**25. Muestra todas las categorías y la cantidad de productos diferentes que tienen asignados. Las categorías vacías deben mostrar 0 (usa `RIGHT JOIN` desde productos hacia categorías).**

SQL

```sql
SELECT c.nombre, COUNT(p.id) AS cantidad_productos
FROM productos p
RIGHT JOIN categorias c ON p.categoria_id = c.id
GROUP BY c.nombre;
```

**Explicación:** Al hacer `RIGHT JOIN`, la tabla `categorias` (a la derecha) se convierte en la principal. Las categorías sin productos generarán nulos en `p.id`, que al ser contados con `COUNT(p.id)` devolverán un `0`.

📝 **Chuleta:** Al igual que en el ejercicio 23, la clave para que salga un "0" en los reportes es contar una columna foránea, nunca usar el comodín `*`.

**26. Calcula el precio medio de los productos agrupados por cada marca comercial.**

SQL

```sql
SELECT marca, ROUND(AVG(precio_actual), 2) AS precio_medio
FROM productos
GROUP BY marca;
```

**Explicación:** Usamos la función de grupo `AVG()` para calcular la media aritmética de los precios, separándolos por marca. Además, le aplicamos `ROUND(..., 2)` para redondear a dos decimales y que quede como formato moneda.

📝 **Chuleta:** `AVG()` ignora automáticamente los valores nulos.

**27. Lista las provincias de nuestros clientes y cuánto dinero han generado en ventas totales cada una.**

SQL

```sql
SELECT c.provincia, SUM(lp.subtotal) AS ventas_totales
FROM clientes c
INNER JOIN pedidos p ON c.id = p.cliente_id
INNER JOIN lineas_pedido lp ON p.id = lp.pedido_id
GROUP BY c.provincia;
```

**Explicación:** Viajamos desde las ventas hasta el cliente para averiguar su provincia, y agrupamos por ella. Así obtenemos la rentabilidad geográfica de la tienda.

📝 **Chuleta:** Puedes agrupar por cualquier columna, no tiene que ser necesariamente un nombre o un ID. Agrupar por provincia o por año es el pan de cada día en el análisis de negocio (BI).

**28. Muestra cada estado posible de los pedidos y cuántos pedidos se encuentran actualmente en cada uno de esos estados.**

SQL

```sql
SELECT estado, COUNT(id) AS cantidad_pedidos
FROM pedidos
GROUP BY estado;
```

**Explicación:** En lugar de usar `DISTINCT` para ver qué estados existen (como hicimos en el ej. 20), el `GROUP BY` nos permite verlos y además contar cuántas filas han caído en cada "cubo".

📝 **Chuleta:** El binomio `SELECT columna, COUNT(...) GROUP BY columna` es la consulta más usada en el mundo para entender cómo se distribuyen los datos de una tabla.

### Bloque 6: Filtrado de Grupos (Joins + GROUP BY + HAVING)

**29. Muestra los clientes que han gastado más de 500 euros en total en nuestra tienda.**

SQL

```sql
SELECT c.nombre_completo, SUM(lp.subtotal) AS total_gastado
FROM clientes c
INNER JOIN pedidos p ON c.id = p.cliente_id
INNER JOIN lineas_pedido lp ON p.id = lp.pedido_id
GROUP BY c.nombre_completo
HAVING SUM(lp.subtotal) > 500;
```

**Explicación:** Es idéntica al ejercicio 21, pero le añadimos un filtro al final. Como la condición de "más de 500" depende de una suma (`SUM`), el filtro debe ir obligatoriamente en el `HAVING`.

📝 **Chuleta:** Regla sagrada: El `WHERE` no entiende de sumas, ni de medias, ni de conteos. Todo lo que lleve una función matemática va al `HAVING`.

**30. Lista las marcas que tienen estrictamente más de 3 productos en nuestro catálogo.**

SQL

```sql
SELECT marca, COUNT(id) AS total_productos
FROM productos
GROUP BY marca
HAVING COUNT(id) > 3;
```

**Explicación:** Agrupamos el inventario por marca, contamos cuántos productos tiene cada una, y desechamos a las marcas pequeñas filtrando los grupos que no superan el número 3.

📝 **Chuleta:** Aunque en el `SELECT` uses un alias (`AS total_productos`), en la cláusula `HAVING` es más seguro volver a escribir la fórmula matemática completa (`COUNT(id)`), ya que Oracle procesa el `HAVING` antes de asignar los alias.

**31. Obtén el nombre de los clientes que han realizado 2 o más pedidos históricos.**

SQL

```sql
SELECT c.nombre_completo, COUNT(p.id) AS total_pedidos
FROM clientes c
INNER JOIN pedidos p ON c.id = p.cliente_id
GROUP BY c.nombre_completo
HAVING COUNT(p.id) >= 2;
```

**Explicación:** Contamos los pedidos de cada cliente y filtramos usando el operador "mayor o igual que" (`>=`).

📝 **Chuleta:** Este tipo de consultas (clientes recurrentes) son vitales para programas de fidelización.

**32. ¿Qué categorías de la tienda han generado más de 1000 euros en ventas totales? Muestra el nombre de la categoría y el importe.**

SQL

```sql
SELECT c.nombre, SUM(lp.subtotal) AS ventas_totales
FROM categorias c
INNER JOIN productos p ON c.id = p.categoria_id
INNER JOIN lineas_pedido lp ON p.id = lp.producto_id
GROUP BY c.nombre
HAVING SUM(lp.subtotal) > 1000;
```

**Explicación:** Agrupamos las ventas subiendo por la jerarquía (línea -> producto -> categoría) y filtramos el resultado acumulado con `HAVING`.

📝 **Chuleta:** Fíjate en el flujo lógico: cruzas (`JOIN`), agrupas (`GROUP BY`) y finalmente limpias lo que no te sirve (`HAVING`).

**33. Muestra los métodos de pago que se han utilizado en más de 5 pedidos distintos.**

SQL

```sql
SELECT metodo_pago, COUNT(id) AS cantidad_pedidos
FROM pedidos
WHERE metodo_pago IS NOT NULL
GROUP BY metodo_pago
HAVING COUNT(id) > 5;
```

**Explicación:** Agrupamos por el método. He añadido un `WHERE` preventivo para que no nos agrupe los pedidos que tienen método nulo (como los cancelados) antes de hacer el conteo.

📝 **Chuleta:** Aquí ves la convivencia perfecta: `WHERE` limpia la basura antes de agrupar, y `HAVING` limpia los grupos pequeños después de sumar/contar.

**34. Lista los productos que han vendido menos de 5 unidades en total. Asegúrate de incluir mediante `LEFT JOIN` aquellos que han vendido 0 unidades.**

SQL

```sql
SELECT p.nombre, NVL(SUM(lp.cantidad), 0) AS unidades_vendidas
FROM productos p
LEFT JOIN lineas_pedido lp ON p.id = lp.producto_id
GROUP BY p.nombre
HAVING NVL(SUM(lp.cantidad), 0) < 5;
```

**Explicación:** Al usar `LEFT JOIN`, los productos sin ventas generan nulos. La función `NVL` los convierte en `0`. Como `0 < 5`, esos productos no vendidos pasarán el filtro del `HAVING` y saldrán en el informe.

📝 **Chuleta:** Si no usaras `NVL` en el `HAVING`, el resultado sería `NULL < 5`, lo cual es "desconocido", y la base de datos no te mostraría los productos con 0 ventas.

**35. Busca a los clientes cuyo "ticket medio" (el promedio del coste total de cada uno de sus pedidos) sea superior a 150 euros.**

SQL

```sql
SELECT c.nombre_completo, ROUND(SUM(lp.subtotal) / COUNT(DISTINCT p.id), 2) AS ticket_medio
FROM clientes c
INNER JOIN pedidos p ON c.id = p.cliente_id
INNER JOIN lineas_pedido lp ON p.id = lp.pedido_id
GROUP BY c.nombre_completo
HAVING (SUM(lp.subtotal) / COUNT(DISTINCT p.id)) > 150;
```


**Explicación:** Para calcular cuánto gasta un cliente "por pedido" de media, dividimos todo lo que ha gastado en su vida (`SUM(subtotal)`) entre el número de pedidos distintos que ha hecho (`COUNT(DISTINCT p.id)`).

📝 **Chuleta:** El `COUNT(DISTINCT columna)` es súper útil dentro de las agrupaciones para no contar duplicados (en este caso, no contar el mismo pedido varias veces si tiene muchos productos dentro).

**36. Muestra las fechas exactas (agrupando por fecha de pedido) en las que la tienda facturó más de 300 euros en un solo día.**

SQL

```sql
SELECT TRUNC(p.fecha_pedido) AS fecha, SUM(lp.subtotal) AS facturacion_diaria
FROM pedidos p
INNER JOIN lineas_pedido lp ON p.id = lp.pedido_id
GROUP BY TRUNC(p.fecha_pedido)
HAVING SUM(lp.subtotal) > 300;
```

**Explicación:** Usamos `TRUNC()` para quitarle las horas a las fechas. Al agrupar por esa fecha truncada, estamos metiendo en el mismo "saco" todas las ventas del día, y luego filtramos los días flojos.

📝 **Chuleta:** Siempre que agrupes por fechas que contengan horas, usa `TRUNC()` (o `TO_CHAR()`) en el `GROUP BY`. Si no lo haces, cada segundo exacto será considerado un grupo diferente.

### Bloque 7: Exclusiones y Uniones Totales (FULL OUTER, IS NULL)

**37. Encuentra a los clientes (nombre y email) que se registraron en la plataforma pero jamás han hecho un pedido (usa `LEFT JOIN` y `WHERE ... IS NULL`).**

SQL

```sql
SELECT c.nombre_completo, c.email
FROM clientes c
LEFT JOIN pedidos p ON c.id = p.cliente_id
WHERE p.id IS NULL;
```

**Explicación:** Clásico Anti-Join. Cruzamos forzando a que salgan todos los clientes. Los que no han comprado devuelven un `NULL` en el ID del pedido. El `WHERE` final busca exactamente esos nulos.

📝 **Chuleta:** Este es el patrón número 1 para auditorías de limpieza de datos en sistemas reales.

**38. Identifica las categorías que actualmente no tienen ningún producto asociado en el inventario.**

SQL

```sql
SELECT c.nombre
FROM categorias c
LEFT JOIN productos p ON c.id = p.categoria_id
WHERE p.id IS NULL;
```

**Explicación:** Misma estrategia. Identificamos elementos "huérfanos" (categorías vacías) forzando el lado izquierdo y filtrando por los nulos de la derecha.

📝 **Chuleta:** Para que funcione un Anti-Join, siempre debes poner en el `WHERE ... IS NULL` la clave primaria (`id`) de la tabla de la derecha.

**39. Muestra los productos (SKU y nombre) que están en el catálogo pero que nunca se han vendido (no aparecen en ninguna línea de pedido).**

SQL

```sql
SELECT p.SKU, p.nombre
FROM productos p
LEFT JOIN lineas_pedido lp ON p.id = lp.producto_id
WHERE lp.pedido_id IS NULL;
```

**Explicación:** Otra aplicación del Anti-Join, esta vez para encontrar stock "muerto" que no tiene salida.

📝 **Chuleta:** A nivel de rendimiento, hacer `LEFT JOIN + IS NULL` suele ser mucho más rápido que usar subconsultas como `WHERE id NOT IN (SELECT...)`.

**40. Realiza un `FULL OUTER JOIN` entre categorías y productos. Observa cómo aparecen tanto los productos sin categoría (si los hubiera) como las categorías sin productos en una sola lista.**

SQL

```sql
SELECT c.nombre AS categoria, p.nombre AS producto
FROM categorias c
FULL OUTER JOIN productos p ON c.id = p.categoria_id;
```

**Explicación:** El `FULL OUTER JOIN` es la fusión de un `LEFT` y un `RIGHT`. Trae todo lo que coincide, más lo que sobra de la izquierda, más lo que sobra de la derecha.

📝 **Chuleta:** Es ideal para reconciliar o comparar dos tablas y ver qué datos faltan en ambos lados a la vez.

**41. Lista el nombre de todos los clientes junto con la fecha de su último pedido (usa `MAX`). Si no tienen pedidos, la fecha debe aparecer nula.**

SQL

```sql
SELECT c.nombre_completo, MAX(p.fecha_pedido) AS ultimo_pedido
FROM clientes c
LEFT JOIN pedidos p ON c.id = p.cliente_id
GROUP BY c.nombre_completo;
```

**Explicación:** El `LEFT JOIN` nos asegura que salen todos. La función agregada `MAX()` sobre una fecha devuelve siempre la más reciente (la más grande cronológicamente).

📝 **Chuleta:** Si quieres la fecha de la primera compra, solo tienes que cambiar `MAX()` por `MIN()`.

**42. Muestra todas las categorías y el total de stock disponible en cada una. Si una categoría está vacía, su stock debe ser nulo o cero.**

SQL

```sql
SELECT c.nombre, NVL(SUM(p.stock), 0) AS stock_total
FROM categorias c
LEFT JOIN productos p ON c.id = p.categoria_id
GROUP BY c.nombre;
```

**Explicación:** Unimos y agrupamos para calcular sumas, pero el `LEFT JOIN` asegura que las categorías vacías no desaparezcan del reporte, asignándoles un estético cero.

📝 **Chuleta:** `SUM` y `NVL` siempre van de la mano en reportes financieros o de inventario.

**43. Obtén el identificador de los pedidos que existen en el sistema pero que se han quedado "huérfanos" (no tienen ninguna línea de pedido asociada).**

SQL

```sql
SELECT p.id
FROM pedidos p
LEFT JOIN lineas_pedido lp ON p.id = lp.pedido_id
WHERE lp.producto_id IS NULL;
```

**Explicación:** Anti-join clásico para detectar un fallo grave de integridad en la base de datos (una cabecera de pedido sin contenido).

📝 **Chuleta:** Gracias a que al crear tu script pusiste `ON DELETE CASCADE`, es muy poco probable que tengas este tipo de inconsistencias.

**44. Muestra una lista de todos los clientes. Si han comprado, añade el ID de sus pedidos. Si no, muestra el cliente con nulos. Utiliza un `RIGHT JOIN` invirtiendo el orden lógico de las tablas.**

SQL

```sql
SELECT c.nombre_completo, p.id AS pedido_id
FROM pedidos p
RIGHT JOIN clientes c ON p.cliente_id = c.id;
```

**Explicación:** Al poner clientes a la derecha y forzar un `RIGHT JOIN`, la base de datos respeta el catálogo completo de clientes, igual que si los hubieras puesto a la izquierda con un `LEFT`.

📝 **Chuleta:** Aunque funciona, acostúmbrate a reescribir tus `RIGHT JOIN` como `LEFT JOIN` invirtiendo las tablas. Es el estándar de facto de la industria.

### Bloque 8: El Combo Completo (JOIN + WHERE + GROUP BY + HAVING)

**45. Muestra el nombre de los clientes de la provincia de 'Barcelona' (`WHERE`) que han gastado más de 200 euros en total (`HAVING`). Cruza 3 tablas.**

SQL

```sql
SELECT c.nombre_completo, SUM(lp.subtotal) AS total_gastado
FROM clientes c
INNER JOIN pedidos p ON c.id = p.cliente_id
INNER JOIN lineas_pedido lp ON p.id = lp.pedido_id
WHERE c.provincia = 'Barcelona'
GROUP BY c.nombre_completo
HAVING SUM(lp.subtotal) > 200;
```

**Explicación:** La estructura definitiva: Filtramos por provincia (fila individual -> `WHERE`), luego agrupamos, calculamos el gasto total, y filtramos a los de poco gasto (grupos matemáticos -> `HAVING`).

📝 **Chuleta:** Este es el orden de ejecución exacto que realiza el motor de la base de datos: `FROM` -> `JOIN` -> `WHERE` -> `GROUP BY` -> `HAVING` -> `SELECT` -> `ORDER BY`.

**46. Lista las categorías que han vendido más de 5 unidades de productos en total, considerando solo los pedidos realizados en el año 2025.**

SQL

```sql
SELECT c.nombre, SUM(lp.cantidad) AS unidades_vendidas
FROM categorias c
INNER JOIN productos pr ON c.id = pr.categoria_id
INNER JOIN lineas_pedido lp ON pr.id = lp.producto_id
INNER JOIN pedidos p ON lp.pedido_id = p.id
WHERE TO_CHAR(p.fecha_pedido, 'YYYY') = '2025'
GROUP BY c.nombre
HAVING SUM(lp.cantidad) > 5;
```

**Explicación:** Cruzamos las 4 tablas (Categoría -> Producto -> Líneas -> Pedidos). Recortamos el bloque enorme quedándonos solo con las ventas de 2025, y luego agregamos y filtramos la suma de unidades.

📝 **Chuleta:** Cuando encadenes tantos JOINs, confía en el mapa de tus claves foráneas. Ve paso a paso desde el origen hasta el destino.

**47. Encuentra a los clientes que han comprado algún producto de la categoría 'Neumáticos' ('NEU') y han gastado, solo en esa categoría, más de 300 euros.**

SQL

```sql
SELECT c.nombre_completo, SUM(lp.subtotal) AS gasto_neumaticos
FROM clientes c
INNER JOIN pedidos p ON c.id = p.cliente_id
INNER JOIN lineas_pedido lp ON p.id = lp.pedido_id
INNER JOIN productos pr ON lp.producto_id = pr.id
WHERE pr.categoria_id = 'NEU'
GROUP BY c.nombre_completo
HAVING SUM(lp.subtotal) > 300;
```

**Explicación:** Al poner `WHERE categoria_id = 'NEU'`, antes de hacer los grupos matemáticos, hemos "borrado" del resultado en memoria cualquier venta de otra categoría. Así, el `SUM()` que hacemos después solo suma los neumáticos.

📝 **Chuleta:** Filtrar inteligentemente en el `WHERE` hace que la base de datos trabaje mucho más rápido, ya que tendrá menos filas que procesar en la agrupación.

**48. Muestra el nombre del producto, la categoría y el total ingresado por sus ventas, solo para productos que cuesten más de 50€ actualmente, y donde el total ingresado supere los 400€.**

SQL

```sql
SELECT pr.nombre AS producto, c.nombre AS categoria, SUM(lp.subtotal) AS total_ingresado
FROM categorias c
INNER JOIN productos pr ON c.id = pr.categoria_id
INNER JOIN lineas_pedido lp ON pr.id = lp.producto_id
WHERE pr.precio_actual > 50
GROUP BY pr.nombre, c.nombre
HAVING SUM(lp.subtotal) > 400;
```

**Explicación:** `WHERE pr.precio_actual > 50` es una condición de fila única (del producto). `HAVING SUM() > 400` es una condición múltiple (de todas sus ventas sumadas).

📝 **Chuleta:** Si pones varias columnas que no sean funciones matemáticas en tu `SELECT` (como `pr.nombre` y `c.nombre`), TODAS ellas tienen que estar obligatoriamente escritas en el `GROUP BY`.

**49. ¿Qué provincias han generado 2 o más pedidos que terminaron en estado 'CANCELADO'?**

SQL

```sql
SELECT c.provincia, COUNT(p.id) AS pedidos_cancelados
FROM clientes c
INNER JOIN pedidos p ON c.id = p.cliente_id
WHERE p.estado = 'CANCELADO'
GROUP BY c.provincia
HAVING COUNT(p.id) >= 2;
```

**Explicación:** Filtramos pedidos individuales cancelados, agrupamos por el territorio del cliente, contamos los "cubos", y descartamos los que tienen menos de 2 incidentes.

📝 **Chuleta:** Usar esto te permite detectar patrones (por ejemplo, "quizás la empresa de mensajería de esa provincia en concreto está fallando").

**50. Calcula el total facturado agrupado por cada método de pago, aplicable solo a los clientes registrados antes de 2025, y mostrando únicamente aquellos métodos que superen los 500 euros facturados. (Implica 4 tablas).**

SQL

```sql
SELECT p.metodo_pago, SUM(lp.subtotal) AS total_facturado
FROM clientes c
INNER JOIN pedidos p ON c.id = p.cliente_id
INNER JOIN lineas_pedido lp ON p.id = lp.pedido_id
WHERE c.fecha_registro < TO_DATE('2025-01-01', 'YYYY-MM-DD')
GROUP BY p.metodo_pago
HAVING SUM(lp.subtotal) > 500;
```

**Explicación:** _(Nota: En realidad solo hacen falta las 3 tablas que contienen la información de registro, método y dinero, pero la estructura lógica de los filtros combinados WHERE/HAVING es lo importante aquí)._

📝 **Chuleta:** Las comparaciones de fechas con `<` o `>` son la mejor forma de establecer barreras temporales seguras sin lidiar con los días exactos en `TO_CHAR`.

### Bloque 9: Producto Cartesiano (Cross Join)

**51. Para preparar una campaña de marketing masiva, genera todas las combinaciones posibles entre los nombres de nuestros clientes y los nombres de las categorías de la tienda (usa `CROSS JOIN`).**

SQL

```sql
SELECT c.nombre_completo, cat.nombre AS categoria
FROM clientes c
CROSS JOIN categorias cat;
```

**Explicación:** El `CROSS JOIN` multiplica el número de filas de la primera tabla por el de la segunda (Si tienes 15 clientes y 15 categorías, devolverá 225 filas de combinaciones directas).

📝 **Chuleta:** Ojo con el `CROSS JOIN`. Como multiplica en lugar de relacionar, si lo haces con tablas grandes (1 millón de filas x 1 millón de filas) bloquearás el servidor intentando generar 1 billón de resultados.

**52. Muestra todas las combinaciones posibles cruzando únicamente a los clientes de 'Madrid' con los productos de la marca 'Brembo'.**

SQL

```sql
SELECT c.nombre_completo, p.nombre AS producto
FROM clientes c
CROSS JOIN productos p
WHERE c.provincia = 'Madrid' AND p.marca = 'Brembo';
```

**Explicación:** Reducimos drásticamente el impacto del producto cartesiano aplicando un `WHERE` que filtra qué porción de la tabla de clientes queremos multiplicar con qué porción de la tabla de productos.

📝 **Chuleta:** Si algún día haces un `INNER JOIN` y te olvidas de poner el `ON`, Oracle ejecutará un producto cartesiano encubierto. ¡Cuidado con eso!
