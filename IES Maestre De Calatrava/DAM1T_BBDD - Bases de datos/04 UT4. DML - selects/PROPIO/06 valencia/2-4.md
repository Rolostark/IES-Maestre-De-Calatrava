**1. Muestra el nombre de cada producto junto con el nombre de su categoría (usa `INNER JOIN` entre `productos` y `categorias`).**

SQL

```sql
SELECT p.nombre AS "Producto", c.nombre AS "Categoria"
FROM productos p
INNER JOIN categorias c ON p.categoria_id = c.id;
```

**Explicación:** Cruzamos la tabla de productos con la de categorías buscando coincidencias exactas. Como la columna `nombre` existe en ambas tablas, usamos los alias `p` y `c` para indicarle a Oracle exactamente de qué tabla queremos extraer cada nombre.

📝 **Chuleta:** Un `INNER JOIN` exige que la condición del `ON` se cumpla en ambas tablas. Si tuvieras un producto sin categoría asignada, no saldría en esta lista.

**2. Lista el identificador de los pedidos y el nombre completo de los clientes que los realizaron.**

SQL

```sql
SELECT p.id AS pedido_id, c.nombre_completo
FROM pedidos p
INNER JOIN clientes c ON p.cliente_id = c.id;
```

**Explicación:** Relacionamos la tabla principal de pedidos con los detalles del cliente utilizando la clave foránea `cliente_id`.

📝 **Chuleta:** Escribir `JOIN` a secas es exactamente lo mismo que escribir `INNER JOIN`. Oracle asume por defecto que quieres una unión estricta.

**3. Muestra el SKU del producto, su nombre y la cantidad vendida en cada línea de pedido.**

SQL

```sql
SELECT p.SKU, p.nombre, lp.cantidad
FROM productos p
INNER JOIN lineas_pedido lp ON p.id = lp.producto_id;
```

**Explicación:** Conectamos el catálogo de productos con el historial de ventas (líneas de pedido) a través del identificador del producto.

📝 **Chuleta:** Un registro de la tabla izquierda (`productos`) puede coincidir con múltiples registros de la tabla derecha (`lineas_pedido`). Por eso verás el mismo producto repetido varias veces si se ha vendido en varios pedidos.

**4. Obtén el nombre de los clientes y la fecha de sus pedidos, pero solo para aquellos pedidos que estén en estado 'ENTREGADO'.**

SQL

```sql
SELECT c.nombre_completo, p.fecha_pedido
FROM clientes c
INNER JOIN pedidos p ON c.id = p.cliente_id
WHERE p.estado = 'ENTREGADO';
```

**Explicación:** Primero "pegamos" las dos tablas mediante el `JOIN` para tener toda la información combinada en memoria, y luego filtramos ese superconjunto resultante usando `WHERE`.

📝 **Chuleta:** El orden lógico es fundamental: primero el `FROM`, luego los `JOIN` con sus condiciones `ON`, y finalmente los filtros `WHERE`.

**5. Genera una lista con el nombre del producto y el nombre de su categoría, pero solo para productos de la marca 'Michelin'.**

SQL

```sql
SELECT p.nombre AS "Producto", c.nombre AS "Categoria"
FROM productos p
INNER JOIN categorias c ON p.categoria_id = c.id
WHERE p.marca = 'Michelin';
```

**Explicación:** Igual que el anterior. La combinación de proyección (alias), cruce de tablas (JOIN) y filtro horizontal (WHERE) nos da exactamente la información solicitada sin duplicados ni ruido.

📝 **Chuleta:** Aunque puedes poner filtros dentro de la cláusula `ON` (ej: `ON p.categoria_id = c.id AND p.marca = 'Michelin'`), las buenas prácticas dictan que el `ON` es para relacionar tablas y el `WHERE` para filtrar datos.

**6. Muestra todos los clientes y la fecha de sus pedidos. Asegúrate de que aparezcan incluso los clientes que nunca han comprado nada (usa `LEFT JOIN` y observa los nulos).**

SQL

```sql
SELECT c.nombre_completo, p.fecha_pedido
FROM clientes c
LEFT JOIN pedidos p ON c.id = p.cliente_id;
```

**Explicación:** Al poner `clientes` a la izquierda del `LEFT JOIN`, le decimos a la base de datos: "Tráeme a todos los clientes sin excepción, y si tienen pedidos, ponme la fecha al lado. Si no tienen, déjalo en blanco (nulo)".

📝 **Chuleta:** Este es el mejor truco para detectar "huérfanos" (clientes sin ventas, productos sin asignar, etc.).

**7. Lista todas las categorías y el nombre de los productos que contienen. Deben aparecer incluso las categorías que no tienen productos asociados.**

SQL

```sql
SELECT c.nombre AS "Categoria", p.nombre AS "Producto"
FROM categorias c
LEFT JOIN productos p ON c.id = p.categoria_id;
```

**Explicación:** Al igual que en el caso anterior, garantizamos que las categorías vacías que creaste (como 'Merchandising' o 'Equipamiento Vintage') no desaparezcan del listado.

📝 **Chuleta:** El orden importa muchísimo. Si hubieras escrito `FROM productos p LEFT JOIN categorias c`, habrías forzado la aparición de todos los productos, no de todas las categorías.

**8. Busca el nombre de los clientes que NO han realizado ningún pedido (pista: usa `LEFT JOIN` y filtra donde el ID del pedido sea `NULL`).**

SQL

```sql
SELECT c.nombre_completo
FROM clientes c
LEFT JOIN pedidos p ON c.id = p.cliente_id
WHERE p.id IS NULL;
```

**Explicación:** Es la evolución del ejercicio 6. Pedimos todos los clientes con sus pedidos, pero luego "cortamos" el resultado quedándonos **únicamente** con aquellas filas donde la base de datos no pudo encontrar una coincidencia en la tabla pedidos (es decir, generó un `NULL`).

📝 **Chuleta:** Esto se conoce como _Anti-Join_ o resta de conjuntos. Es la forma más eficiente de buscar elementos sin actividad o relaciones faltantes.

**9. Muestra todos los productos del catálogo y, si se han vendido, el ID del pedido donde aparecen. Los productos nunca vendidos deben aparecer también.**

SQL

```sql
SELECT p.nombre, lp.pedido_id
FROM productos p
LEFT JOIN lineas_pedido lp ON p.id = lp.producto_id;
```

**Explicación:** Garantizamos la integridad del lado izquierdo (`productos`) para que los ítems con 0 ventas se muestren en el informe con un `NULL` en la columna del pedido.

📝 **Chuleta:** Un producto puede tener múltiples líneas de pedido. El `LEFT JOIN` conservará todas esas ventas y, además, añadirá una fila con valores nulos para los que tengan cero.

**10. Lista todos los pedidos y su método de pago junto al nombre del cliente. Si un cliente no tiene nombre (caso improbable en esta BD), el pedido debería aparecer igual.**

SQL

```sql
SELECT p.id AS pedido_id, p.metodo_pago, c.nombre_completo
FROM pedidos p
LEFT JOIN clientes c ON p.cliente_id = c.id;
```

**Explicación:** Aquí la tabla "importante" que no queremos recortar es `pedidos`. Por eso la ponemos a la izquierda del `LEFT JOIN`.

📝 **Chuleta:** En un entorno bien diseñado (como el tuyo con claves foráneas obligatorias), un `LEFT JOIN` como este devolverá exactamente los mismos resultados que un `INNER JOIN`, ya que no pueden existir pedidos huérfanos sin cliente.

**11. Realiza la misma consulta que el ejercicio 6 (todos los clientes aunque no tengan pedidos), pero esta vez utiliza un `RIGHT JOIN` cambiando el orden de las tablas.**

SQL

```sql
SELECT c.nombre_completo, p.fecha_pedido
FROM pedidos p
RIGHT JOIN clientes c ON p.cliente_id = c.id;
```

**Explicación:** Hace exactamente lo mismo que el ejercicio 6, pero invirtiendo la lógica. Ahora le decimos "conserva todo lo de la tabla que está a tu DERECHA (`clientes`)".

📝 **Chuleta:** Los `RIGHT JOIN` se usan muy poco en el mundo real porque leer el código de derecha a izquierda no es intuitivo. La inmensa mayoría de los programadores reescriben la consulta para usar un `LEFT JOIN`.

**12. Muestra todas las categorías (incluso las vacías) usando un `RIGHT JOIN` con la tabla productos.**

SQL

```sql
SELECT c.nombre AS "Categoria", p.nombre AS "Producto"
FROM productos p
RIGHT JOIN categorias c ON p.categoria_id = c.id;
```

**Explicación:** Exactamente el mismo resultado que el ejercicio 7. Al poner `categorias` a la derecha, forzamos que esa sea la tabla dominante.

📝 **Chuleta:** Recuerda: `A LEFT JOIN B` es matemáticamente idéntico a `B RIGHT JOIN A`.

**13. Obtén una lista de todas las líneas de pedido junto con el nombre del producto, asegurándote de que si un producto hubiera sido borrado, la línea de pedido aparezca con nombre nulo.**

SQL

```sql
SELECT lp.pedido_id, lp.cantidad, lp.precio_unitario, p.nombre AS "Producto"
FROM lineas_pedido lp
LEFT JOIN productos p ON lp.producto_id = p.id;
```

**Explicación:** Para proteger el registro histórico (las ventas), `lineas_pedido` manda (izquierda). Si cruzáramos con `INNER` y alguien borrase un producto, desaparecerían las ventas históricas de los reportes.

📝 **Chuleta:** Esta es una práctica vital en facturación. Nunca debes perder una línea de pedido solo porque el producto origen ya no se vende en la tienda.

**14. Muestra el nombre completo de los clientes y el estado de sus pedidos. Si hay clientes sin pedidos, muestra 'Sin pedidos' en lugar del valor nulo (usa `NVL` o simplemente observa el nulo).**

SQL

```sql
SELECT c.nombre_completo, NVL(p.estado, 'Sin pedidos') AS "Estado Actual"
FROM clientes c
LEFT JOIN pedidos p ON c.id = p.cliente_id;
```

**Explicación:** El `LEFT JOIN` generará un nulo en `p.estado` para los clientes que no han comprado. La función `NVL` intercepta ese nulo y lo reemplaza por un texto amigable.

📝 **Chuleta:** `NVL` (Null Value Logic) es tu mejor amigo para darle formato profesional a los reportes y evitar que aparezcan espacios en blanco feos.

**15. Lista todos los productos y su marca, junto con la cantidad total pedida. Usa `LEFT JOIN` para que los productos con 0 ventas no desaparezcan de la lista.**

SQL

```sql
SELECT p.nombre, p.marca, NVL(SUM(lp.cantidad), 0) AS "Cantidad Total"
FROM productos p
LEFT JOIN lineas_pedido lp ON p.id = lp.producto_id
GROUP BY p.nombre, p.marca;
```

**Explicación:** Unimos productos con sus ventas. Como queremos la "cantidad total", debemos agrupar por el producto. El `LEFT JOIN` nos asegura que el producto salga, y el `NVL` convierte la suma de cantidades "nulas" en un hermoso cero.

📝 **Chuleta:** Cuando mezclas agrupaciones con `LEFT JOIN`, agrupa siempre por las columnas de la tabla dominante (la izquierda).

**16. Une tres tablas: Muestra el nombre del cliente, el ID del pedido y el nombre del producto que compró (doble `INNER JOIN`).**

SQL

```sql
SELECT c.nombre_completo, p.id AS pedido_id, pr.nombre AS "Producto"
FROM clientes c
INNER JOIN pedidos p ON c.id = p.cliente_id
INNER JOIN lineas_pedido lp ON p.id = lp.pedido_id
INNER JOIN productos pr ON lp.producto_id = pr.id;
```

**Explicación:** Para llegar desde el Cliente hasta el Producto, tenemos que saltar por la estructura de tu base de datos: `clientes` -> `pedidos` -> `lineas_pedido` -> `productos`. _(Nota: El enunciado dice "doble INNER JOIN", pero necesitas 3 JOINs para conectar estas 4 tablas)._

📝 **Chuleta:** Puedes encadenar tantos `JOIN` como quieras. La base de datos irá resolviendo las uniones en cascada, de arriba hacia abajo.

**17. Une tres tablas: Muestra el SKU del producto, el nombre de su categoría y el ID del pedido donde se vendió (usa `INNER JOIN`).**

SQL

```sql
SELECT p.SKU, c.nombre AS "Categoria", lp.pedido_id
FROM productos p
INNER JOIN categorias c ON p.categoria_id = c.id
INNER JOIN lineas_pedido lp ON p.id = lp.producto_id;
```

**Explicación:** Partimos de `productos`. Primero cruzamos hacia "arriba" con `categorias` para obtener el nombre de la familia, y luego hacia "abajo" con `lineas_pedido` para ver dónde se ha vendido.

📝 **Chuleta:** El orden en el que escribes los `INNER JOIN` encadenados no afecta al resultado, pero por legibilidad es mejor seguir el flujo lógico de las claves foráneas.

**18. Muestra el nombre de todas las categorías y, si tienen productos vendidos, el ID de la línea de pedido. Usa `LEFT JOIN` encadenados.**

SQL

```sql
SELECT c.nombre AS "Categoria", lp.pedido_id
FROM categorias c
LEFT JOIN productos p ON c.id = p.categoria_id
LEFT JOIN lineas_pedido lp ON p.id = lp.producto_id;
```

**Explicación:** Unimos tres tablas, pero forzando a que `categorias` sea la "jefa" absoluta. Si una categoría no tiene productos, arrastrará nulos. Si tiene productos pero no tienen ventas, arrastrará nulos en el ID del pedido.

📝 **Chuleta:** Si empiezas una cadena con `LEFT JOIN`, asegúrate de usar `LEFT JOIN` en todos los enlaces siguientes si quieres mantener esa obligatoriedad. Si mezclas con un `INNER JOIN` por el camino, anularás el efecto y desaparecerán los nulos.

**19. Obtén el nombre de los clientes y los productos que han comprado, pero solo para clientes de la provincia de 'Valencia' o 'Madrid'.**

SQL

```sql
SELECT c.nombre_completo, pr.nombre AS "Producto"
FROM clientes c
INNER JOIN pedidos p ON c.id = p.cliente_id
INNER JOIN lineas_pedido lp ON p.id = lp.pedido_id
INNER JOIN productos pr ON lp.producto_id = pr.id
WHERE c.provincia IN ('Valencia', 'Madrid');
```

**Explicación:** Volvemos a hacer la ruta completa para conectar al cliente con lo que compró, y le aplicamos un filtro usando el operador de lista `IN` para que busque cualquiera de las dos provincias.

📝 **Chuleta:** El filtro `WHERE` se aplica al bloque de datos gigante una vez que todos los `JOIN` se han terminado de procesar internamente.

**20. Muestra el identificador del pedido, la fecha y el nombre de la categoría de cada producto incluido en dicho pedido.**

SQL

```sql
SELECT p.id AS pedido_id, p.fecha_pedido, c.nombre AS "Categoria"
FROM pedidos p
INNER JOIN lineas_pedido lp ON p.id = lp.pedido_id
INNER JOIN productos pr ON lp.producto_id = pr.id
INNER JOIN categorias c ON pr.categoria_id = c.id;
```

**Explicación:** Otro viaje por las 4 tablas principales. Cruzamos para llegar desde la cabecera del pedido, pasando por el detalle y el producto, hasta aterrizar en la categoría y poder proyectar su nombre.

📝 **Chuleta:** Entender el mapa de relaciones de tu base de datos (qué tabla tiene qué `id` o clave foránea) es el 90% del trabajo. Escribir el `ON` es solo la traducción técnica.
